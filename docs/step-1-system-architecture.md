# Step 1 — System Architecture (ThoKan Cloud)

## 1) Architectural Style

ThoKan Cloud uses a modular monorepo with clear service boundaries:

- `frontend`: Next.js web client (SSR + client interactions)
- `backend`: FastAPI REST service and domain modules
- `database`: PostgreSQL schema and migrations
- `docker`: container and deployment configuration
- `scripts`: bootstrap and operations scripts

The backend is designed as a modular monolith first (faster delivery), with interfaces that allow future extraction into microservices if scale requires it.

## 2) Runtime Components

### Client Layer

- Next.js app for dashboard, file browser, admin panel
- Token-aware API client
- Drag-and-drop uploads with resumable strategy (planned)
- Dark/light theme with design tokens

### API Layer

- FastAPI REST endpoints under `/api/v1`
- Auth: login, refresh, password reset, optional TOTP 2FA
- Files: upload/download/versioning/folders/move/rename/delete
- Sharing: user shares + expiring secure links
- Admin: users/roles/settings/storage insight

### Data Layer

- PostgreSQL for transactional metadata
- Tables for users, roles, files, folders, ACLs, share links, audit logs, versions

### Storage Layer

- Primary local encrypted object storage on disk
- Pluggable storage adapter abstraction:
  - `local` driver (default)
  - `s3` driver (optional)

### Security & Platform Layer

- JWT access + refresh token rotation
- Rate limiting per IP/user
- CSRF protection for cookie-based flows
- Secure headers (HSTS, X-Frame-Options, CSP baseline)
- Audit event pipeline
- File scanning hook before finalizing uploads

## 3) Logical Module Boundaries (Backend)

- `auth`: login/register/reset/2FA/JWT lifecycle
- `users`: profile + admin user management
- `files`: file metadata, folders, version history
- `permissions`: RBAC + folder/file ACL rules
- `sharing`: user shares + secure public links
- `audit`: append-only activity log
- `notifications`: email integration and templates
- `admin`: system settings and reports

## 4) Core Flows

### Upload Flow

1. Client requests upload intent.
2. API validates user permission and target folder ACL.
3. File stream stored in temporary location.
4. Scanning hook executes.
5. File encrypted and persisted to storage driver.
6. Metadata/version row committed in PostgreSQL.
7. Audit event emitted.

### Share Link Flow

1. Authenticated user generates link with expiry and optional password.
2. API issues signed token and stores hashed link secret.
3. Recipient accesses link endpoint.
4. API validates expiry, revocation, access constraints.
5. Download event logged to audit table.

## 5) Deployment Topology

- Reverse proxy (TLS termination, HTTPS redirect, security headers)
- Frontend container
- Backend container
- PostgreSQL container (or managed DB)
- Optional object storage service (S3 compatible)

Internal traffic stays on private docker network. Only reverse proxy exposes public ports.

## 6) Scalability Strategy

- Stateless frontend/backend containers for horizontal scaling
- DB indexing + query optimization for metadata-heavy operations
- Background worker queue planned for:
  - antivirus scanning
  - thumbnail generation
  - async notifications
- CDN support can be added at edge for shared/static assets

## 7) Security-by-Default Principles

- Deny-by-default ACL checks
- Principle of least privilege for roles and service accounts
- Encrypt at rest (file blobs) and in transit (TLS)
- Full audit trail for authentication, file access, admin actions
- Secrets only via environment variables / secret manager

## 8) API Versioning + Contracts

- Base path: `/api/v1`
- OpenAPI generated by FastAPI
- Structured error model:
  - `code`
  - `message`
  - `details`

## 9) Initial Technology Decisions

- Frontend: Next.js 15 + TypeScript + TailwindCSS
- Backend: FastAPI + SQLAlchemy + Alembic
- Auth tokens: JWT (short-lived access, long-lived rotating refresh)
- Password hashing: bcrypt
- Email: SMTP provider abstraction

## 10) Out-of-Scope for Step 1

- Concrete table DDL/migrations
- Endpoint implementation
- Full UI screens
- Docker production tuning details

Those are implemented in subsequent steps.
